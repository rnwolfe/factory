# Factory — Production Usage Audit (last 14 days)

**Window:** 2026-06-06 → 2026-06-20 · **Instance:** `~/.factory-live` · **Author:** Claude (audit run)

## TL;DR

Factory is **working well as a fleet orchestrator**. In 14 days it drove **89 runs across 8 projects**, completing **69 (78%)**, shipping real features and **14 tagged releases** (mabel ×8, backbar ×3, backlot, others), at **$208 in Claude spend** (~$3.87/paid run; codex runs are free via subscription). Run summaries are honest and substantive; agents block on the *right* things (real credentials, spec gates) and never fake completion. The honesty contract is holding.

The problems are concentrated and fixable:

1. **codex + claude-model misconfig = 58% of all failures** (7/12). Silent 3s clean-exit, no output, marked `failed`. Pure config-validation gap.
2. **codex footer non-compliance discards committed work** — 2 runs did ~9.5 min of correct, *auto-committed* work, then got marked `failed` for emitting the wrong fenced block. Real code thrown away.
3. **Inbox/triage "auto-fire on landing" doesn't fire** — the dominant operator complaint (4+ feedback items): feedback/plans land but sit until manually nudged.
4. **Mobile PWA polish** — a cluster of small iOS-Safari/PWA UX papercuts (most already fixed).
5. **Fresh-worktree `bun install` gap** flips typecheck false-red, muddying the quality signal.

## 2. What's going well

- **Run-summary quality is the strongest part of the corpus.** Summaries cite file paths, test counts, migration numbers, and proactively flag stubbed seams ("tts.kokoro is a placeholder", "gated on `AI_GATEWAY_API_KEY`") and autonomous decisions. Examples:
  - backlot: *"routes provider/model strings through the Vercel AI Gateway (ai@6.0.205 — which I had to install; the task note's 'already installed' was wrong)"*
  - mabel: *"Fixed Mabel's 'declares inability before trying' bug… removed the false 'can't browse the web' limit… Full suite (327) passes"*
- **Blocks are legitimate, 6/6.** Agents block only on (a) credentials/hardware they physically can't provision, and (b) spec-gated milestones they correctly refuse to build ahead of (mabel M5: *"the gate is structurally unmet… I intentionally did not build the PWA"*). The operator↔agent dialog is high quality both directions — blocker questions are numbered and name exact files/env vars.
- **`agent_decision` loop works** — 16 mid-run design escalations, all actioned in ~1h avg, capturing substantive architecture calls (schema shapes, provider interfaces, escalation policies).
- **Fleet is shipping.** backlot (50 commits/14d, deepest feature thread), mabel (50 commits, most mature, 8 releases), backbar (41 commits, 3 releases), lodestar (bootstrapped through M0 cleanly). No agent hit a capability wall anywhere.
- **GitHub-issue backend is live** — 13 issue-intake decisions processed (mabel, dwell), 10 actioned / 3 dismissed.

## 3. What's not going well (findings)

### F1 — codex + claude-model pairing silently fast-fails *(highest impact)*
**7 of 12 failures.** codex-agent runs paired with `claude-opus-4-8`/`claude-sonnet-4-6` exit cleanly (`exitCode 0`) in ~3.4s with **zero output** — events show only `iteration_start → session → iteration_end`. The codex CLI accepts the invalid claude model id, no-ops, exits clean → null factory-status parse → `failed`.

Root cause confirmed in code: `apps/daemon/src/workers/submit.ts:230` resolves `effectiveAgent` and `effectiveModel` from **independent** sources and explicitly bets *"a codex model id passed to claude-code (or vice versa) would just bounce."* It doesn't bounce — it fast-fails. The `AgentModelPicker` keeps the pair in sync in the UI, but the submit-time inheritance chain (`input → task.frontmatter → project → settings`) can still assemble a stale mismatch (e.g. project agent flipped to codex while project model still held a claude id — directly tied to feedback task-035, *"model list on task page only includes Claude, not codex"*).

**Fix:** hard-reject incompatible {agent, model} at submit time with a clear operator error (`agentForModel`/registry already knows the mapping). Cheapest highest-value fix in this audit.

### F2 — codex footer non-compliance discards committed work *(highest cost)*
2 runs (factory `t722xnwh` 432s, `woxib1` 131s) did substantial, correct work — implemented snooze mutations across decisions/plans/audits/feedback, ran tests, **auto-committed** — then were marked `failed` because codex emitted a ` ```factory-decision ` block (or no footer) instead of `factory-status`. ~9.5 min of working, committed code thrown to `failed`. One shows `inputTokens: 911489` of real spend marked as failure.

**Fix:** when `exitCode==0` **and** commits exist **and** footer is missing, fall back to a `needs-review`/`uncertain` state instead of hard `failed`. This *softens, never weakens*, the honesty contract — it only rescues the case where work provably landed. Separately, enforce the footer harder on the codex code path.

### F3 — Inbox/triage doesn't auto-fire on landing *(dominant operator complaint)*
Multiple feedback items, same root issue:
- *"When a factory feedback item lands in the inbox, it should automatically triage… Right now I have to send an additional reply to trigger this."* (task-025)
- *"Plans never draft upon landing promoted in inbox (they get stuck on 'drafting') but work fine after I nudge with a comment."* (task-033)
- *"Triage output… is just the raw output of the prompt landing as a comment… not really integrated."* (task-038)
- *"Does the context from a triage session make it into the actual promoted task or plan?"* (task-039)

A whole class of "landed object should self-advance one step, and triage context should flow into the promoted artifact" — currently requires a manual nudge, which leaks into the single attention sink (the inbox). Several resolved; the *pattern* is worth a systemic fix.

### F4 — Mobile PWA papercuts (mostly resolved)
Long worktree paths in run-event log (task-022), iOS-Safari PWA navbar bottom gap (task-023), task-form cursor jump to title on input (task-024), missing pull-to-refresh (task-032), notifications with no project/task context (task-036), inbox cards not showing parent project (task-034), feedback button red/green affordance (task-037). Individually small, collectively the bulk of friction reports. Confirms the phone-first invariant is where lived friction concentrates.

### F5 — Fresh-worktree `bun install` gap flips quality false-red
≥5 runs show `TS2688: Cannot find type definition file for 'bun-types'/'bun'` marking typecheck `exitCode 2` on otherwise-clean code, because the fresh worktree skipped `bun install` before `make typecheck`. One long run self-recovered by running `bun install` mid-run. Also: one quality run couldn't spawn a shell to load `quality.yaml` (`ENOENT posix_spawn 'sh'`). Pollutes the (informational) quality signal.

### F6 — Secondary
- **No agent self-retry** (Gap 5 below) — every transient failure reaches the operator.
- **Intervention-log surface is hollow** — the only two `_intervention-logs/*.log` files are incidental tmux scrollback (operator running `git status`), not structured intervention records. The real intervention loop lives entirely in `blocker_questions`/`operator_context` DB columns; nothing surfaces "stuck → operator did X → outcome" as a queryable primitive.
- **factory project has 0/37 quality reports** — Factory dogfoods itself but has no `quality.yaml` wired, so its own runs skip checks.
- **dwell (production tier, imported today) has no quality gate** — no `quality.yaml`/Makefile, and 0 `factory/run` branches (developed via normal PR flow); Factory's index is currently decorative for it.
- **mabel `complete_task` bug** patched *in-project* (`c9454a0`/`5e91f34`: "matched wrong task — marked 'awning' done"). Fuzzy task-name matching in Factory's `tasks.ts` `complete_task` worth hardening upstream.
- **mabel working tree dirty** (4 files incl. a test) — uncommitted residue worth committing/reverting.
- **fathom dormant** — last commit 2026-05-17; all 5 phase-tasks done, nothing re-fed the queue. Orchestration gap: a completed task list with no re-fill mechanism just stalls silently.

## 4. Proposed work

**Now (bugs, cheap, high-value):**
1. **F1** — reject incompatible {agent, model} at submit with a clear error. *(~1 file, kills 58% of failures.)*
2. **F2** — `exitCode==0 + commits + missing footer → needs-review`, not `failed`. *(Stops discarding committed work.)*
3. **F5** — guarantee `bun install` in fresh worktree before quality checks (or detect bun-types miss → install + retry).

**Next (systemic UX):**
4. **F3** — make landed inbox objects self-advance one step on arrival (auto-triage feedback, auto-draft promoted plans) and thread triage context into the promoted task/plan. This is the single biggest lived-friction theme.
5. **F6** — wire `quality.yaml` for factory itself and dwell.

**Feature proposals (from the patterns):**
- **Single automated self-retry** for environmentally-classifiable failures (wrong model corrected, `bun install` then retry). Most F1/F5 failures would never reach the inbox — directly protects the single attention sink.
- **First-class interventions** — promote the `blocker_questions → operator_context → re-run` chain into a queryable intervention record (it's the richest run-quality signal and currently invisible to any auditor). Retire or repurpose the hollow `_intervention-logs/` files.
- **Queue-empty nudge** — when a project's task list goes all-`done` with auto-advance on, surface a "fathom is out of runway — re-fill or archive?" inbox item instead of silently stalling.
- **Cost-per-project surfacing** — $208/14d is fine, but opus projects are 100% of spend while codex projects are free; a per-project cost chip would let the operator consciously route tier-appropriate work to codex.
